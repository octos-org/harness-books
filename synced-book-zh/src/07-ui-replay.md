# 8. 回放与用户事实：前端只能投影运行时真相

设想一个再普通不过的场景。你交给 agent 一个要跑二十分钟的任务，中途切去回了封邮件，回来时顺手刷新了一下页面。结果进度条归零了，那条“正在重构第 3 个模块”的状态不见了，界面要么卡在一个永远转圈的“运行中”，要么干脆从头开始问你“需要我做什么”。后台其实一切照旧——任务还在那台 worker 上稳稳推进——可你这一侧已经慌了：我那二十分钟是不是白干了？

这一幕的根源，不在后台，而在前端对“真相”的态度。如果第 7 章守的是能力入口，第 8 章守的就是用户表面的真实性。对 harness 来说，前端不是装饰层，而是运行时事实能否被用户正确看见的**最后一跳**——它是第 2 章 §2.3 那个负反馈环里离人最近的那枚传感器。只要这一跳开始自作主张、自己编造状态，系统很快就会掉回第 5.1 到第 5.3 节那些事故里：用户看到的进度和真实进度对不上，而且没人知道哪个才算数。

## 8.1 回放真正要解决的不是“刷新后保留聊天”

很多产品把 replay 理解成“把聊天记录再画一遍”——刷新后，把之前那几屏对话从缓存里捞出来重新渲染。这远远不够，因为一个长任务 agent 的状态，远不止屏幕上那几条消息。真正需要被保留下来的，是任务的生命周期走到了哪个阶段、工具产生了哪些副作用、验证留下了哪些证据、子任务各自交回了什么摘要、当前会话的边界（scope）是什么、终态到底有没有被独立裁决过。这些才是“任务现在处于什么状况”的真身；聊天文本只是它在屏幕上投下的一道影子。

要把这件事做对，得先借用一个后端工程师都熟悉的模式：`事件溯源（event sourcing）`。它的核心主张是，系统的当前状态不该被存成一份可以被随手改写的快照，而应该由一条**只追加、不修改**的事件日志推导出来——想知道“现在是什么状态”，就把这条日志从头（或从最近一个检查点）重放一遍、折叠成当前值。这正是 harness 该有的姿态：后台维护一条权威的事实流，前端的所谓“当前状态”，永远是把这条流重放、投影出来的结果，而不是它自己攒下来的一份私货。

只有站在这个模式上，“可靠的状态气泡”才扛得住真实世界里那些粗暴的打断：浏览器重载、网络重连、切走会话再切回来、页面和后台 worker 各自重启、数据流（通常是一条 SSE 长连接）中断后重新订阅。这些情形的共同点是——它们都会清空前端的内存，却清不掉后台那条事实流。只要前端的状态是从事实流投影出来的，重载后把流再放一遍，看到的就还是同一个真相；可一旦前端把状态攒在了自己的内存里，这一清就真没了。换句话说，replay 缺失或解释不一致时，哪怕后端内部做对了很多事，用户感知到的仍然是一团“随机性”。

## 8.2 Claude Code / Codex 给出的共同约束

两套成熟系统，从不同方向印证了同一条约束。Codex 的 `app-server` 把 `thread/started`、`turn/started`、`item/*`、`turn/completed` 这些通知做成一条统一的事件流——注意它的用意：实时推送给你看的流，和事后回放给你看的历史流，用的是**同一个对象模型**。不存在“直播一套数据结构、回看又是另一套”。Claude Code 则从产品器官的层面说着同一句话：`query.ts` 的执行循环、transcript 的持久化、resume（断点续跑）机制、子 agent 的进度摘要，合起来表明——无论这个 UI 看上去多像一个聊天框，它的本体都不是聊天，而是**一个可恢复执行体的事实投影**。[^replay-truth-ch8]

把这条约束讲到底，就是给前端划一条很硬的边界：UI 最多只配拥有三样东西——把事实流折叠成可显示状态的**投影逻辑**、把状态摆成像素的**展示逻辑**、把用户操作翻译成请求的**交互逻辑**。它唯独不能拥有第四样：**独立事实**。前端可以缓存、可以做动画、可以决定先画哪块后画哪块，但它一旦开始自己判断“这个任务大概已经进到验证阶段了吧”，就等于在事实流之外私设了一个第二真相源——而系统里只要有两个真相源，迟早会打架。

## 8.3 把失败案例翻译成回放设计规则

第 5 章那一连串事故，几乎可以逐条翻译成前端的设计规则。为了不再出现 §5.1、§5.2 那种进度漂移，实时路径和历史路径必须共享同一份任务 schema——直播和回看若各自解释“什么算一个阶段”，迟早会对不上。为了不再出现 §5.3 那种会话污染，session 与 topic 的 scope 必须从事件层一路贯穿到回放层、再贯穿到渲染层，中间任何一段“图省事”地把 scope 丢掉，兄弟会话的状态就会串进当前这个容器。为了不再出现 §5.6 那种操作员盲区，UI 不能只显示一个干巴巴的“当前状态”，还得把最近一个动作、最近一条证据、最近一次失败原因一并摆出来——因为人要判断“它是不是真的在好好干活”，靠的从来不是那个状态词，而是这些旁证。

这几条合起来，其实只指向一件事：**前端里不应该再存在第二套生命周期解释器**，也不应该允许某些状态气泡只来自本地推断。前端可以做缓存、动画和布局，但不能凭空“发明”出一个阶段。它发明的每一个阶段，都是一条迟早要和后台对质、并且必输的影子真相。

## 8.4 把回放当成可靠性的测试面

一旦把“前端只能投影、不能发明”定成架构原则，测试策略也得跟着升级——因为这类问题恰恰是单元测试照不到的死角，它们只在“刷新、重连、切会话”这些动作发生时才现形。值得被写成门禁（CI gate）的断言，大体是这样几类：任务快照接口（如 `/api/sessions/:id/tasks`）给出的状态，必须和把 SSE 流回放一遍得到的状态在生命周期上完全一致；刷新前后、重连前后、切走再切回前后，用户看到的必须是同一份任务事实；历史回放绝不能把兄弟 session 的状态带进当前容器；子 agent 的摘要、主产物的状态、验证证据，在刷新之后必须依然可见；数据流中断后，UI 靠 replay 重建出来的状态，必须和恢复直推后继续推送的状态严丝合缝、不打架。

这些断言读起来琐碎，但它们正是“用户事实层”是否真的存在的试金石。只要它们还没被提升成合并/发布的门禁，团队就还停留在“实现了一个前端”，而没有实现一个**真正可信的用户事实层**——区别在于，前者刷新一下就可能骗人，后者怎么刷都说同一句实话。

[^replay-truth-ch8]: 本章在此处综合 Codex `app-server` 中 thread / turn / item 的统一事件流设计，以及 Claude Code transcript、resume 与 agent summary 机制，说明 UI replay 必须建立在统一的后端事实模型之上；对应第 21 章参考文献 21、24。
