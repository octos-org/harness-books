# 6. 回放与用户事实：前端只能投影运行时真相

如果第 5 章守的是能力入口，第 6 章守的就是用户表面的真实性。对 harness 来说，前端不是装饰层，而是运行时事实能否被用户正确看见的最后一跳。只要这一跳开始自作主张，系统很快就会掉回第 3.1 到第 3.3 节那些事故里。

## 6.1 回放真正要解决的不是“刷新后保留聊天”

很多产品把 replay 理解成“把聊天记录再画一遍”。这远远不够。真正需要被保留下来的，不只是消息文本，而是任务生命周期、工具副作用、验证证据、子任务摘要、会话 scope 和终态裁决。

因此，可靠的状态气泡至少要扛住这些情形：

- 浏览器重载；
- 重新连接；
- 切走会话后再切回来；
- 页面和 worker 各自重启；
- 部分 stream 中断后重新订阅。

如果 replay 缺失或解释不一致，即使后端内部做对了很多事，用户看到的仍然会是“随机性”。

## 6.2 Claude Code / Codex 给出的共同约束

Codex 的 `app-server` 把 `thread/started`、`turn/started`、`item/*`、`turn/completed` 这些通知做成统一事件流，这是在告诉我们：实时流和历史流必须共享一个对象模型。Claude Code 的 `query.ts`、transcript 持久化、resume 和子 agent 进度摘要，则从产品器官层面说明同一件事：无论 UI 看起来多像聊天，本体都不是聊天，而是一个可恢复执行体的事实投影。[^replay-truth-ch6]

换句话说，UI 最多拥有三样东西：

- 投影逻辑；
- 展示逻辑；
- 交互逻辑。

它不能拥有第四样东西：独立事实。

## 6.3 失败案例怎样变成回放设计规则

第 3 章可以直接翻译成三条回放规则：

1. 为了避免 3.1 和 3.2 的进度漂移，实时路径和历史路径必须共享同一份任务 schema。
2. 为了避免 3.3 的会话污染，session / topic scope 必须从事件层一直贯穿到回放层和渲染层。
3. 为了避免 3.6 的操作员盲区，UI 不只要显示“当前状态”，还要显示最近行动、最近证据和最近失败原因。

因此，前端里不应再存在第二套生命周期解释器，也不应允许某些气泡状态只来自本地推断。前端可以做缓存、动画和布局，但不能发明阶段。

## 6.4 实战落地：把回放当作可靠性测试面

回放架构一旦定下来，测试策略也应跟着改：

- 断言 `/api/sessions/:id/tasks` 快照与 SSE 回放在生命周期上完全一致；
- 断言刷新前后、重连前后、切换会话前后，用户看到的是同一任务事实；
- 断言历史回放不会把兄弟 session 的状态带进当前容器；
- 断言子 agent 摘要、主产物状态和验证证据在刷新后仍然可见；
- 断言 stream 中断后，UI 通过 replay 重建的状态与实时继续推送的状态不冲突。

只要这些断言没有成为门禁，团队就还只是“实现了前端”，还没有实现真正的用户事实层。

[^replay-truth-ch6]: 本章在此处综合 Codex `app-server` 中 thread / turn / item 的统一事件流设计，以及 Claude Code transcript、resume 与 agent summary 机制，说明 UI replay 必须建立在统一的后端事实模型之上；对应第 19 章参考文献 21、24。
