# 6. UI 回放不是前端功能，而是可靠性功能

如果第 5 章解决的是外部接入的一致性，第 6 章解决的就是用户表面的真实性。对 harness 来说，前端不是装饰层，而是运行时事实能否被用户正确看见的最后一跳。

可靠的状态气泡必须承受：

- 浏览器重载
- 重新连接
- 切走会话后再切回来
- 部分 stream 中断

如果 replay 缺失或不一致，即使后端正确，用户也会看到“随机性”。

设计规则：

- 回放路径和实时路径必须共享规范事件/任务 schema
- UI 中不允许第二套解释层
- 门禁测试必须断言任务 API 和 SSE 回放之间的一致性

Codex 的 `app-server` 之所以值得反复提，是因为它把 `thread/started`、`turn/started`、`item/*`、`turn/completed` 这些通知明确做成统一事件流；Claude Code 的 `query.ts`、transcript 持久化和子 agent 进度摘要，则说明另一个方向的同样结论：无论 UI 再怎么丰富，回放和实时都必须基于同一组后端事实。成熟系统不会让前端自己发明生命周期解释器。

UI 回放的难点在于它横跨两种时间：实时流和历史回放。实时流让用户看到“正在发生什么”，历史回放让用户在刷新、断线、切会话后仍然看到“已经发生过什么”。如果这两条路径使用不同解释逻辑，系统迟早会出现“刚才显示 running，刷新后变 pending”这类体验。

所以 UI 不应该拥有独立状态真相。它只负责把后端规范事实投影出来。前端可以做缓存、动画和布局，但不能发明生命周期。
