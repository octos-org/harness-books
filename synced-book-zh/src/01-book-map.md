# 0. 全书地图与阅读指南

这份单文件书稿是 Octos 驾驭工程的合并版综合稿。它保留既有叙事主线，并把 ZIP 源书的前置说明、阅读指南、附录和参考文献折叠进来，让整个论证可以在一个地方读完。

## 0.1 先抓住这本书的主线

为了避免章节多了以后主线发散，可以先把全书逻辑压成六步：

1. LLM 有容量上限，但工程上真正重要的是有效工作区间，而不是 brochure 上的最大 context。
2. 一旦任务跨越长时间、多工具、多会话，质量、成本、时延和状态一致性会一起恶化，所以不能把 prompt 当系统。
3. OpenAI、Anthropic 这类前沿团队之所以能把 agent 用到生产，不是因为 prompt 更花，而是因为有了 harness、验证、回放、知识分层和操作员控制面。
4. Octos 当前遇到的多数事故，本质上都能还原为同一类问题：系统把“类似聊天的输出”误当成“运行时事实”。
5. 因此解法必须围绕 session、harness、tools、verification 四支柱展开，并落到事件 ABI、UI replay、operator dashboard、mini-fleet 门禁、sub-agent/swarm 编排这些具体机制上。
6. 最后还要把这些机制沉淀成原则、路线图、清单和边界说明，避免系统只会“偶尔成功”，却不能稳定交付。

## 0.2 这本书适合谁

- 需要稳定运行时事实、而不是演示级行为的 Octos 维护者
- 需要进度、产物、验证契约的 app-skill 作者
- 需要不靠聊天文本猜测失败模式的平台操作员

## 0.3 推荐阅读顺序

- 线性路径：按顺序阅读第 1 到第 14 节
- Layer 1 路径：重点读第 1、2、3、5、10、11、13、14 节
- Layer 2 路径：重点读第 1、2、3、6、7、8、9、12、13 节
- 按角色路径：使用文末附录中的负责人、架构师、CTO 和研究者阅读路线

## 0.4 这个合并版新增了什么

- 面向 Octos 读者的紧凑前置说明
- 十篇奠基论文和文章的源文本附录
- 四根 harness 支柱的自查矩阵
- 贯穿全书的核心术语表
- 按角色组织的阅读指南和研究轨迹

核心弧线保持不变：第 1 节提出控制系统视角，第 3 节把 Octos 事故转化为失败类别，第 4 节和第 7 节把这些失败转化为运行时架构，后续章节区分 harness 能解决什么，以及哪些仍然依赖模型质量、治理和操作纪律。
