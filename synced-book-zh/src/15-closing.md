# 14. 结语：下一阶段的工程姿态

到这里，整本书的主线也就闭合了：从长上下文和长任务的根因出发，经过 Claude Code / Codex 等成熟样本、OpenClaw / Hermes 的相邻对照、一组真实失败案例的反证、运行时架构、组织扩展与治理边界，最后落回可执行的工程姿态。

把 harness 工作当作产品可靠性工程，而不是抽象框架建设。

对任何还在从 demo 走向系统的 agent 团队，更合理的胜利模式是：

1. 识别具体用户可见失败类别
2. 编码显式契约不变量
3. 把运行时/UI/操作员表面绑定到同一个事实模型
4. 合并前在小型 fleet 线上门禁上证明
5. 之后再扩展抽象和外部平台表面

这就是从“今天看起来很聪明”走向“每天都能正确交付”的路径。
