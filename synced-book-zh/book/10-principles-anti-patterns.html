9. 从抽象到原则：Harness Engineering 的设计法则

前面 1 到 8 章已经把根因、样本、失败、架构和协作面铺开了。第 9 章要做的不是再发明新部件，而是把这些观察压缩成一套更高层、但仍然能落地的设计法则。

这里最值得借鉴的，不是某个仓库的代码排版，而是 Fowler / Thoughtworks 那条线对 harness engineering 的抽象：不要把 agent 的成功寄托在一次 prompt 奇迹上，而要把任务状态、反馈回路、工具边界和验证证据做成系统外部、可检查、可演化的控制层。1

9.1 第一原则：把模型能力和控制系统分开

Claude Code 和 Codex 都很强，但它们真正可迁移的启发不在于“某个 prompt 怎么写”，而在于都在做同一件事：把模型生成能力与控制系统分层。

模型负责提出候选动作、生成候选产物、总结局部信息；控制系统负责持久化事实、调度动作、绑定权限、裁决终态、组织回放和支撑运营。只要这两层没有被切开，团队最终一定会重新回到第 3 章那类事故里。

9.2 九条可迁移原则

把第 2.7 节那九个架构维度进一步压缩，可以得到九条可迁移原则：

  1. 契约状态必须持久化在 prompt / 聊天上下文之外。
  2. 生命周期必须单调、公开，并在 API / UI / operator 表面保持一致。
  3. 所有外部能力都必须进入统一调度面，而不是各自维持私有状态机。
  4. 主产物必须显式归属；交付不能依赖文件名或启发式猜测。
  5. ready / failed 必须由验证器和证据裁决，而不是由生成器自报完成。
  6. session / topic / worktree scope 必须被当成硬边界，而不是软约定。
  7. 回放、摘要和 dashboard 必须都从同一条事实流派生。
  8. sub-agent / swarm 的价值来自角色化、隔离、汇总和验证，而不是单纯并行。
  9. 每一个发布声明都必须绑定门禁证据和事故归档，而不是轶事式成功。

这九条原则的价值在于，它们不依赖 Claude Code 用 TypeScript 还是 Codex 用 Rust，也不依赖团队是否采用 MCP、worktree 或某个特定 UI。它们描述的是控制系统必须守住的结构性边界。

9.3 必须禁止的反模式

与上面九条原则一一对应，下面这些反模式必须被明确禁止：

  1. 只有 prompt 的契约，例如“agent 应该记得发送文件”。
  2. 从 UI 反推事实，例如“气泡显示 done，所以任务完成”。
  3. 用 stderr 文本、自由格式日志或缓存快照充当状态协议。
  4. 用文件名、最近修改时间或目录启发式猜主产物。
  5. 让尽力而为式验证器不影响终态成功。
  6. 让子 agent 直接宣布系统终态，或直接污染父任务事实。
  7. 让 dashboard 指标和运行时事实流脱节。
  8. 在协议变更时只改实现，不改 fixture、门禁、E2E 和文档。

这些反模式的共同点，是把“系统说真话”退化成“系统看起来差不多在工作”。一开始它们都节省代码，后来都会以事故成本把节省加倍收回来。

9.4 短循环和长循环

所有 agent 都有短循环：生成、行动、观察，再生成。驾驭工程真正关心的是长循环：事故如何进入 wiki，wiki 如何晋升成 skill，skill 如何进入 AGENTS.md 或 docs，docs 中的约束如何再沉成 CI、validator 和 dashboard 信号。2

没有长循环，agent 每次都像从零开始的实习生。长循环跑起来之后，组织会把一次失败变成长期免疫力。

9.5 知识和 Skill 的晋升纪律

不是所有经验都应该写进 AGENTS.md。更稳妥的治理方式是 promotion / demotion：重复出现、跨任务稳定、能被验证的经验才晋升；过时、低频、无法验证或引起误导的经验要降级或删除。3

Skill 也不是 prompt 集合。一个合格 skill 至少要有触发条件、输入输出契约、执行步骤和验证方式。对这类系统来说，skill 的价值不在“让模型多知道一点”，而在“把组织经验变成可执行、可检查的流程”。

Codex 的 AGENTS.md 文档化和分层作用域设计，说明了什么应该常驻、什么应该按需加载；Claude Code 的 loadAgentsDir.ts、skills、hooks、memory scope 则说明了知识成熟后怎样沉进 agent 定义、技能包和能力边界。知识治理不是单独的文档工作，它会反过来塑造工具面、角色面和验证面。

9.6 反身 harness 的边界

未来 harness 会越来越多地修改自己：更新文档、生成测试、补充验证器、建议 schema 迁移。这是趋势，但反身能力必须分级。低风险变更可以自动化；影响协议、权限、验证独立性和合规目标的变更，必须保留人工决策。不能把目标函数、权限边界和验证器独立性，也交给提案者自己定义。
