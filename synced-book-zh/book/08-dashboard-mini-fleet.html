驾驭工程

9. 操作员控制面与发布真实性

凌晨两点，值班的人盯着一屏仪表盘：错误率平稳、延迟正常、吞吐没掉，一片祥和的绿色。可与此同时，一个高价值客户的任务已经卡死了四十分钟——它静静地停在某个阶段，既没报错、也没超时，只是不再往前走。聚合指标永远不会为它报警，因为单独一个任务卡住，根本撼动不了平均值。值班的人能从这屏图表里读出系统“最近很忙”，却读不出“这一单此刻到底怎么了、我该重试还是该叫醒开发者”。

到第 8 章为止，我们回答的是“用户如何看到真相”；第 9 章回答的是另一半问题：“操作员如何足够快地理解真相，并据此做发布决策”。这一步极其关键，因为很多团队走到这里会犯同一个错误——系统内部已经攒了不少事件和日志，于是想当然地以为“可观测性差不多有了”。但要分清两个东西：可观测性（observability）回答的是“系统整体健不健康”，靠的是指标和日志的聚合；而 harness 真正需要的是控制面（control plane），它回答的是“这一个任务此刻卡在哪、为什么、我下一步该按哪个按钮”。前者是一堆仪表，后者是一个驾驶舱。日志堆得再高，也堆不出一个驾驶舱。

9.1 操作员真正要回答的是哪几个问题

判断一个操作员视图够不够格，标准很朴素：把一个真实的人放到它面前，他能不能在几十秒内回答出那几个最要命的问题——哪个任务卡住了，卡在哪个阶段，它最近一个动作是什么，是哪个 validator 失败了，主产物现在在哪儿，这究竟是任务自身的问题还是某个子 agent、某个外部工具拖累的，以及最终那个决定：该重试、该回滚、该继续等，还是该升级给开发者。

只盯着计数器，这些问题一个都答不上来。计数器能告诉你“最近很忙”“失败率升高了”，却答不出“这一次具体为什么错”——而操作员需要的恰恰是后者。这正是第 5.6 节那种操作员盲区的由来：系统不是没有数据，而是数据停在了“聚合统计”这一层，没有被进一步组织成“一个任务的来龙去脉”。盲区不是信息太少造成的，而是信息没被加工成可据以行动的形状。

9.2 Claude Code / Codex 给了什么操作员线索

两套系统在这件事上给的提示，朴素却管用。Claude Code 的 agentSummary.ts、diskOutput.ts 和 resume 机制透露出一个很实用的取向：不该逼操作员去直接读原始 transcript 或原始日志。一个人在排障时要的不是一万行流水，而是一份摘要、一个“最近做了什么”、一个产物在哪、这活归谁、以及一个“从这儿接着跑”的恢复入口。Codex 则用 Thread / Turn / Item 的对象拆分，提供了更适合当 dashboard 地基的数据骨架——它把“历史”“正在执行的一步”“产生的副作用”分到了不同的层上，于是这些东西天然就能被重新组织成一段操作员读得懂的叙事，而不是糊成一团的文本。1

把两者合起来，能提炼出一条很稳的经验：操作员视图不该只消费 metrics，它必须消费从事实流派生出来的叙事结构。指标是给系统体检用的，叙事才是给人做决策用的；控制面的本职，是把那条只追加的事实流（第 8 章），按“一个任务的视角”重新折叠成人能在几十秒内看懂的故事。

9.3 仪表盘要绑死在同一条事实流上

一个靠谱的 dashboard，展示的不该是一堆好看的曲线，而该是从事实流里投影出来的几个固定切面，并且每个切面都能在那条流里找到出处。生命周期切面回答“现在在哪个阶段、什么时候进的、有没有违反单调性（比如阶段不该倒退却倒退了）”；动作切面回答“最近调了什么工具、派了什么子任务、出了什么摘要”；产物切面回答“主产物在哪条路径、上次改于何时、过没过验证”；证据切面回答“是哪个 validator 失败、证据落在哪、错误归哪一类、有没有超时”；作用域切面回答“这事属于哪个 session / topic / worktree、归谁所有”；运营切面则回答“重试了几次、有没有孤儿任务、有没有串扰信号、要不要人工介入”。

这么设计的目的不是让 dashboard 更华丽，而是让它和第 6 章那张运行时总图一一对齐——图上有的维度，盘上就该有对应的切面；盘上有的状态，流里就该找得到来源。这条对齐关系是有纪律含义的：只要 dashboard 上出现一个“只在图表里有、却在事实流里找不到出处”的状态，系统就又凭空多了一条影子真相——和第 8 章前端“发明阶段”是同一类病，只不过这次发明者换成了监控层。

9.4 小型 fleet 不是测试的补充，而是发布的事实

单元测试和集成测试当然必要，但对 harness 远远不够。原因在第 2 章其实已经埋下：那些最难缠的错误是时序型、多表面、跨 session 的——它们依赖真实的并发、真实的重连、真实的多主机时序，而这些恰恰是单机单元测试构造不出来的。它们只在真实的 worker、真实的重载路径、真实的会话切换里才肯现身。

所以需要一个小型 fleet——可以把它理解成一支袖珍的“金丝雀（canary）”机群：在正式放量之前，先用一小撮真实主机、真实流量，把系统该有的不变量跑一遍。它该被当成发布真实性的最低证明面，而不是可有可无的加分项。具体说，它至少要在两台以上主机上跑脚本化门禁，至少覆盖两个 session、一次重载、一次重连、一次子任务或外部能力调用；它必须强制输出显式的诊断类别，而不是笼统一句 failed（因为“失败了”帮不了任何人定位）；而且一旦某条不变量被破坏，它要能直接挡住合并或发布，而不是记一条告警了事。

最值得被门禁化的那些不变量，几乎原封不动地来自第 5 章那座事故博物馆：必需阶段不能缺失，阶段序列不能非单调，终态不能在验证之前就被宣布成功，会话 scope 不能泄漏，research / background 这类任务不能被重复触发，子 agent 的摘要不能和主任务的事实失联。这些都是用真实事故换来的边界——把它们写成门禁，等于让系统替你记住那些教训。

9.5 让事故博物馆回流成控制面的免疫力

第 5.7 节讲“失败模式博物馆”时，重点是归档——把每一次有价值的翻车，连同它的根因和复盘，存成一条可查的记录。到了第 9 章，重点要从归档变成回流：每一个高价值的失败模式，最终都应该被映射成某样具体的东西——一个 dashboard 信号、一段门禁脚本，或一条 operator runbook（写明“看到这个现象，就照这几步处理”的操作手册）。

这一步常被省掉，但它恰恰是复盘有没有白做的分水岭。只有完成了这次回流，一个事故故事才真正变成了控制面的免疫力——下次同类问题再冒头，它要么被门禁直接挡在合并之前，要么在 dashboard 上一眼可见、并配着现成的处置步骤；而不是写完一份漂亮的复盘文档，归档、然后等着它换个马甲再来一次。控制面的成熟度，某种意义上就等于“有多少条事故已经回流成了免疫力”。
